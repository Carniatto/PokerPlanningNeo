# Lessons Learned - 2025-12-09

## Context
Refining the mobile PWA experience for Poker Planning Neo, focusing on scrolling issues, touch interactions, and visual consistency between Host and Player views.

## Mobile Touch Interactions
- **Sticky Hover**: The `:hover` pseudo-class persists on touch devices until another element is tapped. This causes buttons to look "stuck" in a pressed state.
    - **Fix**: Wrap hover styles in `@media (hover: hover)` so they only apply to pointer devices.
    - **Code Example**:
      ```css
      @media (hover: hover) {
        .btn:hover { ... }
      }
      ```

## CSS Layout & Overflow
- **Nested Scrolling Risks**: Assigning `overflow-y: auto` to a child container (like a grid) within a parent that also scrolls can create "scroll traps" and clip visual effects (shadows, transforms).
    - **Takeaway**: For mobile vertical flows, prefer `display: contents` for wrappers or simply letting the document flow naturally within a single top-level scroll container (`height: 100dvh`).
- **Input vs. Text Padding**: Browsers add default padding to `<input>` elements. When trying to align a static text header (e.g., `<h1>`) with an input field, precise manual padding (e.g., `padding: 0.2rem 0.5rem`) is often required to match the browser's default input styling.

## Angular Build Budgets
- **Production Builds**: Angular's default build budgets (e.g., 4KB for component styles) are quite strict for modern, rich UI themes (gradients, shadows).
    - **Solution**: Check `angular.json` under `configurations.production.budgets`. Increasing `anyComponentStyle` to ~20KB and `initial` to ~2MB is a reasonable adjustment for feature-rich applications.

## Grid Layouts
- **Consistency**: When sharing a mental model (e.g., "A Deck of Cards") between two different views (Host vs Player), use identical Grid definitions (`grid-template-columns: repeat(4, 1fr)`) even if the surrounding containers differ. This preserves the user's spatial memory of the deck.
